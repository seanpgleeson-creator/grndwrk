## 1. Stack & Project Setup

| Layer | Choice | Notes |
|-------|--------|-------|
| Framework | Next.js (App Router) | API routes in `app/api/` |
| ORM | Prisma 7 + PostgreSQL | `@prisma/adapter-pg` (`PrismaPg`); `DATABASE_URL` in `prisma.config.ts` + runtime env |
| AI | Anthropic Claude API | **Phase 2.** Abstract behind `lib/ai/claude.ts` — not present in Phase 1 |
| Validation | Zod | All AI responses and request bodies |

### Required environment variables

```
DATABASE_URL=          # Required. PostgreSQL URL, e.g. postgresql://user:pass@host/db?sslmode=require (Neon)
ANTHROPIC_API_KEY=     # Phase 2 — sk-ant-...
NODE_ENV=              # Set by Next.js / platform (development | production)
```

### Deployment (Vercel + Neon)

- **App shell:** `app/(app)/layout.tsx` sets `export const dynamic = "force-dynamic"` so authenticated routes are not statically prerendered at build time (avoids Prisma errors when `DATABASE_URL` is unreachable during local `npm run build`).
- **Build:** `vercel.json` runs `npm run vercel-build` → `prisma migrate deploy && next build`.
- **If you see P3019** (datasource provider does not match `migration_lock.toml`): the migration history was created for a different DB engine. Remove `prisma/migrations/`, run `prisma migrate dev` against a Postgres database to regenerate, commit the new folder. If the Neon database already has unrelated tables from another project, reset that database (or use a fresh Neon database) before applying grndwrk migrations.
- **Do not** commit real secrets; only `DATABASE_URL` in Vercel env vars.

### PrismaClient singleton (`lib/prisma.ts`)

Prisma 7 uses a driver adapter for PostgreSQL:

```typescript
import { PrismaClient } from "@prisma/client";
import { PrismaPg } from "@prisma/adapter-pg";

function createPrismaClient(): PrismaClient {
  const dbUrl = process.env.DATABASE_URL;
  if (!dbUrl) throw new Error("DATABASE_URL is not set.");
  const adapter = new PrismaPg({ connectionString: dbUrl });
  return new PrismaClient({ adapter });
}

const globalForPrisma = globalThis as unknown as { prisma: PrismaClient };

export const prisma = globalForPrisma.prisma ?? createPrismaClient();

if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;
```


## 4. AI Service Layer

**Status:** Phase 2. The repository does not yet include `lib/ai/` or Anthropic SDK wiring. Phase 1 API routes that call AI return HTTP **501** with a not-implemented message.

### 4.1 `lib/ai/claude.ts` — single entry point (Phase 2)

```typescript
import Anthropic from "@anthropic-ai/sdk";
import { prisma } from "@/lib/prisma";

const client = new Anthropic(); // reads ANTHROPIC_API_KEY from env

export interface ClaudeCallOptions {
  system: string;
  user: string;
  maxTokens?: number;
}

export async function callClaude(options: ClaudeCallOptions): Promise<string> {
  const { system, user, maxTokens = 4096 } = options;

  let lastError: Error | null = null;

  for (let attempt = 0; attempt < 3; attempt++) {
    try {
      const response = await client.messages.create({
        model: "claude-sonnet-4-5",
        max_tokens: maxTokens,
        system,
        messages: [{ role: "user", content: user }],
      });

      const block = response.content[0];
      if (block.type !== "text") throw new Error("Unexpected response type");
      return block.text;
    } catch (err) {
      lastError = err as Error;
      if (attempt < 2) await sleep(1000 * (attempt + 1));
    }
  }

  throw lastError;
}

// Convenience: inject positioning context automatically from UserProfile.
// Use this in all route handlers instead of callClaude directly.
export async function callClaudeWithProfile(options: {
  systemSuffix: string;  // prompt-specific system instructions appended after profile context
  user: string;
  maxTokens?: number;
}): Promise<string> {
  const profile = await prisma.userProfile.findUnique({
    where: { id: "singleton" },
    select: { positioning_statement: true, narrative_pillars: true },
  });

  const profileContext = profile
    ? `User positioning statement:\n${profile.positioning_statement ?? "(not set)"}\n\nNarrative pillars:\n${JSON.stringify(profile.narrative_pillars)}`
    : "";

  return callClaude({
    system: `${profileContext}\n\n${options.systemSuffix}`,
    user: options.user,
    maxTokens: options.maxTokens,
  });
}

function sleep(ms: number) {
  return new Promise((resolve) => setTimeout(resolve, ms));
}
```

### 4.2 Prompt files (`lib/ai/prompts/`)

Each file exports:
1. A function that builds the prompt string given inputs
2. A Zod schema for validating the JSON response
3. A typed parse function that validates and returns the structured result

**Pattern for all prompt files:**

```typescript
// lib/ai/prompts/cmf.ts
import { z } from "zod";

export const CmfResponseSchema = z.object({
  domain:    z.object({ score: z.number().min(1).max(10), rationale: z.string() }),
  stage:     z.object({ score: z.number().min(1).max(10), rationale: z.string() }),
  scope:     z.object({ score: z.number().min(1).max(10), rationale: z.string() }),
  strategic: z.object({ score: z.number().min(1).max(10), rationale: z.string() }),
  narrative: z.object({ score: z.number().min(1).max(10), rationale: z.string() }),
  resume_gap_analysis:        z.string(),
  recommended_tweaks:         z.array(z.string()),
  application_recommendation: z.enum(["prioritize", "proceed", "marginal", "skip"]),
});

export type CmfResponse = z.infer<typeof CmfResponseSchema>;

export function buildCmfSystemPrompt(weights: Record<string, number>): string {
  return `You are a career strategy analyst. Score the candidate's fit for a role across five dimensions using the provided candidate profile as context.

CMF dimension weights (how much each dimension matters to this candidate):
${JSON.stringify(weights, null, 2)}

Return valid JSON matching this exact shape:
{
  "domain":    { "score": 1-10, "rationale": "..." },
  "stage":     { "score": 1-10, "rationale": "..." },
  "scope":     { "score": 1-10, "rationale": "..." },
  "strategic": { "score": 1-10, "rationale": "..." },
  "narrative": { "score": 1-10, "rationale": "..." },
  "resume_gap_analysis": "...",
  "recommended_tweaks": ["...", "..."],
  "application_recommendation": "prioritize" | "proceed" | "marginal" | "skip"
}

Composite CMF score = sum(dimension_score * weight/100). The application_recommendation thresholds: 8–10 = prioritize, 6–7 = proceed, 4–5 = marginal, below 4 = skip.`;
}

export function buildCmfUserPrompt(params: {
  jdText: string;
  resumeParsed: unknown;
  companyResearch?: string;
}): string {
  return `Job description:\n${params.jdText}\n\nParsed resume:\n${JSON.stringify(params.resumeParsed, null, 2)}\n\n${params.companyResearch ? `Company research:\n${params.companyResearch}` : ""}`;
}

export function parseCmfResponse(raw: string): CmfResponse {
  const json = JSON.parse(raw);
  return CmfResponseSchema.parse(json);
}
```

**All prompt contracts:**

| File | Input | Output schema |
|------|-------|---------------|
| `resumeParse.ts` | `resume_raw: string` | `{ experience: [], skills: [], education: [] }` |
| `cmf.ts` | `jd_text`, `resume_parsed`, `company_research?`, `cmf_weights` | Per-dimension scores + rationale + gap analysis + recommendation |
| `earnings.ts` | `transcript: string`, `company_name?: string` | `{ hiring_signals[], strategic_priorities[], problem_areas[], outreach_trigger_score, suggested_follow_ups[] }` |
| `narrativeCheck.ts` | `content: string` | `{ consistency_score (1–5), assessments[], explanation }` |
| `companyBrief.ts` | Company profile + signals + user profile | Sections: why_company, why_now, value_proposition, proof_points[], the_ask |
| `roleBrief.ts` | JD text + company profile + user profile + company brief | Sections: fit_summary, contribution_narrative, differentiated_value, proof_points[] |
| `coverLetter.ts` | Role brief + narrative pillars + company type | Full cover letter text |
| `outreachDraft.ts` | Signal card + user profile + company profile | Short outreach message text |

**Narrative consistency check** runs as a secondary call after any content generation. Route handlers call it after writing the primary AI output, then include `consistency_score` and `explanation` in the API response. If `consistency_score < 3`, the client shows a yellow warning banner.

## 5. Draft/Edited Pattern

Applies to every piece of AI-generated content:
- `CompanyPositioningBrief`: `draft`, `edited`
- `RolePositioningBrief`: `draft`, `edited`
- `Opportunity.materials.cover_letter`: `{ draft, edited }`
- Outreach drafts (stored in `OutreachRecord.message_summary` — the user always edits before sending)

**Rules:**

| Rule | Detail |
|------|--------|
| `draft` is write-once | Set at AI generation time. Never updated by subsequent `PATCH` calls. |
| `edited` starts null | Only set when the user explicitly saves edits. |
| Read precedence | Always read `edited ?? draft`. Never show `draft` directly if `edited` exists. |
| Reset | `POST .../reset` sets `edited = null`. Next read falls back to `draft`. |
| `PATCH` routes | Only ever write to `edited` (and individual section fields). Reject attempts to write to `draft` from the client. |

## 6. Singleton UserProfile Pattern

- The `UserProfile` table will have exactly one row with `id = "singleton"`.
- First `GET /api/profile`: if no row exists, return `{ status: 404, onboarding_required: true }`. The client middleware redirects to `/profile/setup`.
- All `POST /api/profile` calls use Prisma `upsert` with `where: { id: "singleton" }`.
- **Onboarding complete check:** `positioning_statement !== null && Array.isArray(narrative_pillars) && narrative_pillars.length > 0`. Both steps 1 and 2 of the onboarding wizard must be completed. Steps 3 (CMF weights) and 4 (comp targets) have pre-filled defaults and are not blocking.
- On first run, `prisma/seed.ts` creates the singleton row with all defaults so `GET /api/profile` always returns a record after seeding.

## 7. Funnel Computation (Dashboard)

Computed at query time in `GET /api/dashboard`. Not stored. Reads `Opportunity[]` with related `RolePositioningBrief`.

```
Monitoring         = count where status IN (Watching, Preparing)
Positioned         = count where status IN (Watching, Preparing)
                       AND brief.completed_at IS NOT NULL
                       AND outreach_sent = false
Applied/Outreach   = count where status = Applied OR outreach_sent = true
In Process         = count where status = InProcess
Outcome            = count where status = Closed
```

Note: an Opportunity can satisfy multiple funnel stages simultaneously (e.g. outreach_sent but still Watching). The funnel counts are intentionally overlapping — they reflect activity flags, not a strict state machine. The UI should present them as a flow, not a mutually exclusive pipeline.

**Priority Action Queue logic** (top 5, for `GET /api/dashboard`):

Ranked by urgency tier (highest first), then recency within tier:

1. EarningsSignal with `outreach_trigger_score >= 4` and no OutreachRecord to that company in last 14 days
2. Opportunity in `InProcess` with no OutreachRecord in last 7 days
3. Contact at a Tier 1 Company with `last_contact` older than 30 days (or null)
4. Opportunity where `cmf_score` is null and `status != Closed`
5. CompanyPositioningBrief where `completed_at` is null and company is Tier 1
6. Opportunity in `Applied` status with no follow-up OutreachRecord in last 14 days

Return the top 5 items with: `{ type, label, company_name?, action, entity_id, entity_type }`.

## 8. Comp Snapshot Strategy

The `Opportunity.comp_snapshot` JSON field stores a point-in-time copy of compensation data. It is a snapshot — not a live reference to `CompBenchmark` — so it is stable even if benchmark data is later refreshed or deleted.

**When to write the snapshot:**

1. When an Opportunity is created (`POST /api/opportunities`)
2. When `PATCH /api/opportunities/[id]` is called and `role_family` or `level` has changed
3. On-demand via `PATCH /api/opportunities/[id]` with `{ refresh_comp_snapshot: true }`

**Snapshot computation:**

```typescript
async function buildCompSnapshot(
  companyId: string,
  roleFamily: string,
  level: string | null,
): Promise<CompSnapshot | null> {
  const benchmark = await prisma.compBenchmark.findFirst({
    where: { company_id: companyId, role_family: roleFamily, level: level ?? undefined },
  });
  if (!benchmark) return null;

  const profile = await prisma.userProfile.findUnique({
    where: { id: "singleton" },
    select: { comp_target: true },
  });

  const stale = isOlderThan180Days(benchmark.fetched_at);
  const meetsTarget = benchmark.total_high != null && profile?.comp_target
    ? benchmark.total_high >= (profile.comp_target as any).minimum
    : null;

  return {
    base_low: benchmark.base_low,
    base_high: benchmark.base_high,
    total_low: benchmark.total_low,
    total_high: benchmark.total_high,
    source: benchmark.source,
    fetched_at: benchmark.fetched_at.toISOString(),
    stale,
    meets_target: meetsTarget, // true | false | null (unknown)
  };
}
```

## 9. Error Handling Conventions

All API routes follow a consistent error response shape:

```typescript
// Success
{ data: T }

// Error
{ error: string, message: string, retryable?: boolean, fields?: Record<string, string> }
```

| Scenario | HTTP Status | `error` value | Notes |
|----------|-------------|---------------|-------|
| AI call failure | 502 | `"ai_failure"` | Include `retryable: true`. Never discard user data. |
| Levels.fyi fetch failure | 200 | — | Return `{ data: null, fallback: true }`. Not treated as an error by the client. |
| Validation error | 422 | `"validation_error"` | Include `fields` with per-field messages. |
| Not found | 404 | `"not_found"` | |
| CMF weights don't sum to 100 | 422 | `"invalid_cmf_weights"` | Include the current sum in the message. |
| Unauthenticated (Phase 4) | 401 | `"unauthorized"` | |

**Never swallow errors silently.** Always `console.error` in development. In production, log to your error tracking service (add in Phase 2+).

**Long-running AI calls** (CMF scoring, brief generation, cover letter): These may take 5–15 seconds. Do not set a short server timeout. In Next.js App Router, set `export const maxDuration = 60` on route segments that call AI.

```typescript
// app/api/opportunities/[id]/cmf/route.ts
export const maxDuration = 60; // seconds — required for long AI calls
```

## Appendix: CMF Score Calculation

The composite score is computed in application code after receiving AI dimension scores:

```typescript
function computeCmfScore(
  breakdown: CmfBreakdown,
  weights: CmfWeights,
): number {
  const dims = ["domain", "stage", "scope", "strategic", "narrative"] as const;
  const total = dims.reduce((sum, dim) => {
    return sum + breakdown[dim].score * (weights[dim] / 100);
  }, 0);
  return Math.round(total * 10) / 10; // one decimal place
}
```

This is stored in `Opportunity.cmf_score` alongside the full `cmf_breakdown` JSON.

## Appendix: Company Warmth Derivation

`Company.warmth` is not stored in the database. It is computed from the company's related contacts whenever a company is returned from the API:

```typescript
function deriveCompanyWarmth(contacts: { warmth: ContactWarmth }[]): ContactWarmth | null {
  if (contacts.length === 0) return null;
  if (contacts.some((c) => c.warmth === "Hot")) return "Hot";
  if (contacts.some((c) => c.warmth === "Warm")) return "Warm";
  return "Cold";
}
```

Include this computation in any route that returns Company data.
